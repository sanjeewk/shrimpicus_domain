# CODEX.md

This file provides guidance to Codex when working with code in this repository.

## Commands

The user appears to use the package-local virtualenv inside `shrimpicus/.venv`, not a repo-root `.venv`.

Activate and run the main surfaces:

```bash
source shrimpicus/.venv/bin/activate
shrimpicus        # Discord bot, entrypoint -> shrimpicus.main:main
shrimpicus-web    # Flask web UI, entrypoint -> shrimpicus.web.app:main
shrimpicus-mcp    # MCP server, entrypoint -> shrimpicus.mcp_server:main
```

Alternative direct module entrypoints:

```bash
python -m shrimpicus.main
python -m shrimpicus.web.app
python -m shrimpicus.mcp_server
```

Quick verification:

```bash
python3 -m py_compile shrimpicus/db.py shrimpicus/web/app.py
```

Requires Python `>=3.11`. There is no formal test suite or linter configured in the repo.

Configuration is loaded by `pydantic-settings` from `<repo-root>/.env`. `DISCORD_BOT_TOKEN` is required for the bot. SQLite is the default database (`./data/shrimpicus.db`), but `DATABASE_URL` can switch the app to PostgreSQL.

## Architecture

Shrimpicus has three user-facing entrypoints that share most of the same core logic:

```text
Settings
  -> Database
  -> NotionService / ObsidianJournal / OllamaClient
  -> AssistantService
       -> Discord bot (bot.py + scheduler.py)
       -> MCP server (mcp_server.py + tools.py)
       -> Web app (web/app.py, partly shared and partly raw SQL)
```

Important boundaries:

- `shrimpicus/main.py` wires the Discord bot runtime together.
- `shrimpicus/assistant.py` is the main business-logic layer for reminders, todos, habits, birthdays, journaling, and free-text assistant behavior.
- `shrimpicus/db.py` owns schema creation, migrations, and the main persistence API.
- `shrimpicus/bot.py` defines prefix commands plus the free-text Discord routing path.
- `shrimpicus/scheduler.py` runs reminder and birthday jobs after Discord login.
- `shrimpicus/tools.py` is the shared tool registry used by the LLM tool-calling loop and by the MCP server.
- `shrimpicus/web/app.py` is the Flask UI. It does not fully go through `AssistantService`; it still contains direct SQL and its own lightweight migration helpers.

### Core package: `shrimpicus/`

- `__init__.py`: package marker.
- `__main__.py`: forwards `python -m shrimpicus` to `main()`.
- `main.py`: main async bot entrypoint; loads settings, initializes services, creates the bot, and starts it.
- `config.py`: `Settings` model and env-file loading.
- `db.py`: schema creation, SQLite migrations, PostgreSQL support, and all persistence helpers.
- `assistant.py`: main application service; categorizes todos, builds RAG context, runs the tool-calling loop, and exposes user actions.
- `bot.py`: Discord command handlers, free-text message routing, transcription hook, and Discord account linking command.
- `scheduler.py`: APScheduler jobs for due reminders and daily birthday checks; includes the reminder check-in button view.
- `tools.py`: JSON-schema tool registry and dispatch layer shared by the assistant and MCP server.
- `mcp_server.py`: stdio MCP server that reuses the same assistant tools.

### Integrations and support modules

- `auth.py`: Argon2 password hashing and basic username/password validation.
- `notion.py`: optional Notion page creation for todos.
- `obsidian.py`: optional append-only journal writer for an Obsidian vault file.
- `ollama.py`: LLM client abstraction supporting local Ollama and hosted OpenRouter.
- `transcribe.py`: optional audio transcription via `faster-whisper`.
- `social_notifications.py`: sends Discord celebrations when users complete goals in shared groups.

### Web app: `shrimpicus/web/`

- `__init__.py`: package marker.
- `app.py`: Flask app, auth flows, board/habits/social pages, API endpoints, and web-specific migration helpers.

## Working Notes

- The in-use virtualenv is `shrimpicus/.venv/bin/activate`.
- Always initialize the database with `db.init()` before using it directly.
- The repo supports both SQLite and PostgreSQL, but the web app still has SQLite-oriented direct SQL in several places.
- `user_id` is the real account owner. `chat_id` is usually a Discord delivery target or a compatibility field from earlier single-user behavior.
- Discord account linking is done through a one-time web-generated code consumed by `!link <code>`.
- Scheduler startup belongs in `bot.py:on_ready`, not in `main.py`, because it depends on the active Discord event loop.
- Voice support is optional and disabled unless the whisper dependency chain is installed.

## Gotchas

- Schema compatibility matters. Older SQLite databases may be missing columns added by later features, so changes in `db.py` and `web/app.py` need to preserve lazy migration behavior.
- `web/app.py` duplicates parts of schema/bootstrap logic instead of fully delegating to `Database`, so DB changes often need parallel updates there.
- Free-text assistant behavior is split between rule-based parsing in `AssistantService` and model-backed tool calling through `OllamaClient` + `tools.py`.
- The bot, web app, and MCP server share concepts but not a single end-to-end request path, so changes should be checked across all three surfaces.
